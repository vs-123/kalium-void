= KALIUM VOID

*Kalium Void* is my own personal spin-off of Void Linux. 

== FEATURES

* **HYBRID REPOSITORY** {em} Utilises `kalium-packages` for certain core components in conjunction with official Void Linux repositories
* **MODIFIED BASE** {em} Utilises `kalium-base-files` in place of standard `base-files`
* **UNIQUE IDENTITY** {em} Distinct branding and identification separate from upstream Void Linux whilst maintaining Void Linux heritage
* **NIX INTEGRATION** {em} Nix package manager is configured to be used
* **PRE-INSTALLED SOFTWARE** {em} Includes `xz`, `unzip`, `firefox`, `gpg`, `git`, `neovim` and additional core utilities.
* **TYPOGRAPHY** {em} Fonts link:https://fonts.google.com/specimen/Tinos[Tinos] and link:https://fonts.google.com/specimen/Libertinus+Mono[Libertinus Mono] are baked in
* **DISPLAY STACK** {em} Uses X.Org/X11 display server with Openbox window manager
* **CUSTOM BRANDING** {em} Has its own unique logo
* **DEFAULT SHELL** {em} `zsh` is configured as the default system shell
* **BOOTLOADER** {em} GRUB is utilised by default
* **GUIDED INSTALLATION** {em} An installer script is included for system configuration and automated dotfile deployment with `chezmoi`

== HOW TO INSTALL

Assuming you've booted into the live ISO, you may install Kalium Void in one of two ways {em} <<installer-method,GUIDED METHOD>> or <<manual-method,MANUAL METHOD>>

[[installer-method]]
=== GUIDED METHOD

An installer script `kalium-installer.sh` is provided at `~/scripts/kalium-installer.sh`. You may execute `kalium-installer.sh` directly as an administrator and follow the on-screen instructions.

[source,shell]
----
sudo ~/scripts/kalium-installer.sh
----

[[manual-method]]
=== MANUAL METHOD

* Make disk and format and mount

[source,shell]
----
cfdisk
# /dev/sda1   1G  (EFI)
# /dev/sda2   12G (SWAP)
# /dev/sda3   1G  (ROOT PARTITION)

mkfs.vfat /dev/sda1
mkfs.ext4 /dev/sda3
mkswap    /dev/sda2

mount --mkdir /dev/sda3 /mnt
mount --mkdir /dev/sda1 /mnt/boot/efi
swapon /dev/sda2
----

* Instead of manually installing packages, clone the live session

[source,shell]
----
sudo rsync -axHAWXS --numeric-ids --info=progress2 / /mnt

# `-x` -- STAY ON ONE FILE SYSTEM 
#         (DON'T TRY TO COPY `/DEV` OR `/PROC` VIRTUAL FILES).
#
# `-aHAWXS` -- PRESERVES EVERYTHING SUCH AS 
#              PERMISSIONS, HARD LINKS, SYMLINKS 
#              AND EXTENDED ATTRIBUTES.
----


* Generate fstab

[source,shell]
----
xgenfstab -U /mnt > outputs.txt
# open outputs.txt in an editor, copy its content 
# and *append* it to /mnt/etc/fstab

# OR

xgenfstab -U /mnt >> /mnt/etc/fstab
----

* Mount these `/dev` `/sys` `/proc`

[source,shell]
----
mount --rbind /dev /mnt/dev
mount --rbind /sys /mnt/sys
mount --rbind /proc /mnt/proc

# OR

for i in dev sys proc; do; mount --rbind /$i /mnt/$i; done
----

* `chroot` and setup stuff

[source,shell]
----
chroot /mnt /bin/zsh

# SETUP
passwd   # SET ROOT PASSWORD
useradd -m -G wheel,audio,video,storage,users USER_NAME   # ADD NEW USER `USER_NAME`
passwd USER_NAME

# INSTALL GRUB
grub-install --target=arm64-efi --efi-directory=/boot/efi --bootloader-id="Kalium Void"

xbps-reconfigure -fa

# SANITY CHECK FOR GRUB
ls /boot/grub/
grep menuentry /boot/grub/grub.cfg
----

* Exit `chroot`

[source,shell]
----
exit
umount -R /mnt
----

* Poweroff, eject the ISO and boot up.

* *THE END*

== LORE

Kalium Void was not born out of a desire to create yet another distro, it was born out of a cycle of persistent frustration and a pursuit of frictionless installation.

For a long time, the standard procedure for installing setting up Void Linux involved a repetitive, mundane task {em} performing a chroot installation , followed by the repetitive task of installing a specific set of core packages, manually configuring the window manager, and fighting with system defaults. Although I usually prefer chroot installation over void-installer, this remains the same both ways, i.e. the volume of manual intervention required to reach a functional, personalised state became quite annoying.

My first attempt to mitigate this was writing a basic `install.sh` script. It wasn't designed to install the OS itself, but rather it was to automate the post-installation phase. Its only job was to pull in my favourite packages and change my default shell. While this helped, it quickly revealed a new bottleneck {em} dotfile management.

I initially tried bundling my dots into a directory right next to my script and rewrote `install.sh` to copy them to their appropriate places, however this manual management grew too complex and became quite brittle to maintain. I eventually discovered `chezmoi`. My installation procedure had now simplified to just: manual chroot install, run `install.sh` and finally `chezmoi`.

Despite these improvements, the initial installation remained a chore to me. I wished to automate the chroot process as well, for which I wrote a secondary script, but it introduced its own irony {em} I'd have to manually install git and clone the repository onto the live ISO just to begin the "automated" process. The workflow had become yet another multi-step chain: boot, install git, clone, run script, reboot, run install script, and finally run chezmoi.

All I wanted was my Void Linux to be ready the moment the live media booted. I finally gave up on scripts and dived into the internals of Void Linux's build tools. As I learnt more, my goal had shifted from writing the best script to was to baking my environment directly into the installation media. 

After extensive research, iteration and lots of trial and error, I managed to build a custom image that moved the logic of the install.sh script directly into the system's base layer. This had transformed my installation workflow into a more streamlined procedure: boot the live image, do chroot installation and apply configurations. From that point on, Kalium Void has continued evolving to what it is right now.

== MORE INFORMATION

*Kalium Void* is a personal distribution engineered and tailored specifically to cater to my own preferences and configurations. As a result, it is not recommended for general use since the design choices, baked-in packages and system defaults are optimised for my specific workflow rather than broad compatibility or standard user expectations. Those who remain curious however are free to install and experiment with the system at their own risk.
